Part 1 — Model Deployment
Section 01 Production Readiness — Real-Time vs. Batch Inference
The core engineering challenge of this project is not building a model that works — it is building one that works at a price a real SaaS product can sustain. At 1,000 users each submitting 100 resume optimization requests per month, an API-based GPT-4 solution would cost approximately $8,000/month in inference alone — a figure that makes a viable product economically impossible at early-stage scale. Our self-hosted fine-tuned Qwen 4B brings that number down to $300–500/month, a reduction of more than 94%.
Achieving this requires a deliberate deployment architecture. The system operates in two distinct inference modes depending on who is asking and when they need an answer.
Real-time inference — primary user-facing mode

When a candidate submits their resume and a job description through the web interface, they expect a result within seconds — not minutes. The system handles this through synchronous, streaming inference on the self-hosted Qwen 4B model. The user sees the tailored resume being written token-by-token, which dramatically improves perceived responsiveness even when total generation time runs 15–40 seconds for a full 10K-token prompt.

At 4-bit quantization, the model requires only 2–3GB of VRAM, meaning a single A10G or RTX 4090 GPU can serve multiple concurrent users simultaneously — a critical requirement for SaaS scalability that the original API-based approach could never offer at this cost point.

Batch inference — secondary enterprise mode

Batch mode has two roles in this system. First, it was already used during data preparation: the four Gemini batch request files (521 + 521 + 523 + 1,044 requests) generated all 1,530 teacher outputs that the student model learned from — batch API pricing cut that data generation cost by approximately 40–50% versus real-time API calls.

In production, batch mode enables an enterprise recruiting use case: a firm wants to optimize 500 candidate resumes against a single job description overnight. All jobs queue to the same self-hosted GPU, process sequentially during off-hours, and return results by morning — at a marginal infrastructure cost near zero beyond the existing GPU rental.

Bridge How Deployment and XAI Talk to Each Other
🔗
Connecting deployment design to explainability findings
The XAI analysis in Part 2 is not a standalone academic exercise — it is the specification for what the production system must monitor. Because shared token ratio is the strongest feature driver of output quality, the deployment architecture logs this metric for every incoming request. This creates a real-time fairness dashboard: if the system observes that candidates with shared token ratios below 0.08 consistently receive outputs with fewer than 15 skill items, that is a live signal that the model is underserving low-overlap candidates — career-changers, non-native speakers, recent graduates — and retraining or prompt intervention is needed.
Similarly, because the bias audit identified a file-type quality gap between PDF and DOCX outputs, the monitoring dashboard tracks mean tailored_skill_item_count and tailored_experience_bullet_count broken down by filetype in production. A persistent, statistically significant gap between PDF and DOCX users is a deployment red flag, not just a research finding. The XAI work done in Section 7 is the foundation that makes responsible deployment possible — without it, bias would be invisible until it caused harm.
Part 2 — Explainable AI (XAI)
Section 04 Feature Importance — Ranking Variables by Impact
Methodological note — why traditional SHAP cannot be applied here

Standard SHAP (SHapley Additive exPlanations) and LIME are designed for models that produce a single scalar output — a predicted class, a probability, or a regression value. Our fine-tuned Qwen 4B is a generative language model: it produces a sequence of tokens, not a single number. There is no scalar loss surface to differentiate with respect to individual input features in the way shap.TreeExplainer or shap.LinearExplainer requires.

This is not a gap in our work — it is an open and actively researched challenge across the entire field of LLM explainability. Our approach: proxy-based feature importance — measuring how each engineered input feature correlates with measurable output quality metrics (content block count, skill item count, experience bullet count, schema validity) derived from the 1,530-sample EDA dataset. This is the most defensible and interpretable XAI method available for generative models.

Section 05 Global Explanation — How the Model Thinks Across All Candidates
Imagine you are a senior recruiter who has reviewed thousands of resumes. Over time, you develop an instinct: you stop reading every word and instead scan for keyword alignment between what the candidate has done and what the job requires. You also notice that longer, richer resumes give you more to work with when tailoring a pitch for a specific role. And no matter how rushed you are, you always fill in the same sections — name, summary, experience, skills — because that structure is non-negotiable.
This is precisely the logic our fine-tuned Qwen 4B SLM learned. Trained on 1,530 resume–job description pairs with Gemini-generated teacher outputs as targets, the model internalized three global behaviors it applies consistently to every candidate.
The three rules the model learned — globally

Rule 1 — Match the vocabulary first. The model's most influential signal is how much the candidate's existing resume vocabulary overlaps with the job description. Across 1,530 training examples the mean shared token ratio was 0.1568 — meaning the average candidate already had about 15–16% of the job description's vocabulary somewhere in their resume. When this overlap is higher, the model produces denser, more targeted outputs.

Rule 2 — Scale depth to length. The model learned that a longer resume is a richer canvas. The mean resume was 1,876 words — more than four times the average job description length of 447 words. Longer resumes produced more bullets (mean: 25.49) and more skills (mean: 29.95). The model does not invent content — it proportionally restructures what the candidate already gave it.

Rule 3 — Never break the schema. Regardless of input quality, the model always outputs a valid JSON structure with all required fields: name, contact, summary, experience, education, skills. Every one of the 1,530 teacher outputs demonstrated this discipline, and the student model absorbed it unconditionally through behavioral distillation.

"In plain terms: the model behaves like a structured editor, not a writer. It does not create a new resume from scratch — it reorganizes and realigns the candidate's existing material toward the specific language and priorities of the job description, then locks everything into a consistent schema. This is critical for legal and ethical defensibility in hiring contexts — every bullet and skill traces back to something the candidate originally wrote."

Plain-language explanation — what this means for John D. specifically

John received a richer-than-average tailored resume for two compounding reasons. First, his resume already contained 23.1% of the job description's vocabulary — nearly one and a half times the dataset average of 15.7%. When the job description asked for "PyTorch, AWS, REST APIs, cross-functional leadership," those terms were already present in John's source resume. The model had strong anchors to build around and did not need to bridge a vocabulary gap.

Second, John's resume was 25% longer than the average candidate, giving the model more raw material to reorganize. The combination of high alignment signal and abundant source content explains why his output had 31 experience bullets and 38 skill items versus the dataset means of 25.49 and 29.95 respectively. The job description length had essentially no effect — at 512 words it was close enough to the dataset mean of 447 words to contribute nothing meaningful above baseline.

In business terms: John was the ideal candidate for this model. His resume and the job description already spoke the same technical language, and he had enough career history for the model to tell a rich, structured story on his behalf.

Section 07 Bias & Fairness Audit — Who the Model Helps, and Who It Hurts
Technical performance metrics tell us whether the model works. They do not tell us whether it works equally for everyone. A resume optimizer that consistently produces richer outputs for some candidates and thinner outputs for others is not a neutral tool — it is a system that amplifies existing advantages at the exact moment a candidate's career trajectory is at stake. The following audit examines four subgroups where the model's behavior, grounded directly in EDA statistics, creates measurable and ethically significant disparities.
📄
Bias 1 — File format penalizes certain candidates
Medium risk

Your EDA found a clear length disparity by file type: .doc files have a median of 2,008 words, .docx files 1,631 words, and .pdf files just 1,231 words. Since resume word count is the second most influential feature, PDF candidates systematically receive thinner tailored resumes through no fault of their professional history. The dataset is also heavily imbalanced — 72.4% of training examples are .docx, while only 16.1% are .pdf and 11.4% are .doc — meaning the model has seen far fewer examples of PDF-format resumes.

So what: PDF is the format most commonly used by candidates without access to Microsoft Word — including international applicants, candidates from lower-income backgrounds, and older professionals. If those candidates receive structurally weaker optimized resumes, the model is quietly reinforcing a socioeconomic barrier that has nothing to do with their qualifications.
⚠️
Bias 2 — Token overlap rewards insiders, penalizes career-changers
High risk — most critical finding

The model's most important feature — shared token ratio (mean 0.1568) — measures how much of the job description's vocabulary already appears in the candidate's resume. Candidates who score well above this get richer outputs. The critical problem is who naturally has high token overlap and who does not. Experienced professionals applying to similar roles will have high overlap. Career-changers, recent graduates, non-native English speakers, and candidates from non-technical backgrounds will have low overlap — and these are precisely the people who need resume optimization most.

Your EDA also identified that job description bigrams include boilerplate phrases like "jobs united," "data provided," and "resume match" — Selenium scraping artifacts. If these tokens appear in the overlap calculation, candidates with generic corporate language score artificially higher while candidates with genuinely specialized skills score lower.

So what: A career-changer from marketing targeting a data science role will have low token overlap — not because they lack potential, but because they have not yet used the industry's vocabulary. The model produces a thin output for them. A senior data scientist making a lateral move gets a rich output. The system helps those who already look like the job description and fails those who are trying to become it. This is the opposite of equitable optimization.
🌍
Bias 3 — Resume length norms reflect one cultural standard
Medium risk

The dataset's mean of 1,876 words reflects North American professional resume culture. This is not universal — European resumes are shorter and more structured, international CVs vary widely, and recent graduates often have 300–700 word resumes by necessity. The model was trained on a corpus dominated by North American .docx resumes and learned what a "normal" resume looks like from that lens. Any candidate who deviates from that norm is implicitly penalized.

So what: Candidates from different cultural backgrounds or career stages receive thinner outputs not because their qualifications are weaker, but because their document structure does not match the training distribution — a form of structural discrimination embedded in the data.
🔤
Bias 4 — Lexical diversity advantages native English speakers
Medium risk

Resume lexical diversity is the fourth most influential feature. Higher diversity correlates with richer outputs. But lexical diversity in English text is strongly influenced by language fluency and writing confidence. Non-native English speakers often write more repetitively — not because their skills are narrower, but because they rely on a safer vocabulary set to avoid errors. The model interprets this as a signal for thinner outputs when in reality it is a signal of language learning, not professional inadequacy.

So what: A highly qualified engineer whose first language is Mandarin, Hindi, or Spanish may produce a resume with lower measured lexical diversity than a less qualified native English speaker. The model generates a richer tailored resume for the native speaker — a form of indirect language discrimination invisible in standard performance metrics but real in its effect on hiring outcomes.

Recommended mitigations
1
Replace lexical token overlap with semantic embedding similarity. Compute cosine similarity between sentence-transformer embeddings of the resume and job description instead of exact token matching. A candidate who wrote "led a team of engineers" and a job requiring "team leadership" would score high semantically but near-zero on token overlap.
Directly addresses Bias 2 and Bias 4 — career-changers and non-native speakers are no longer penalized for vocabulary mismatch when their experience is genuinely relevant.
2
Strip job-board boilerplate before computing any overlap metric. Remove "jobs united," "data provided," "resume match," and similar Selenium scraping artifacts identified in your EDA before computing shared_token_ratio.
Makes the model's primary feature reflect genuine professional fit rather than scraping noise.
3
Augment training data with short resumes and PDF-format candidates. The current 72.4% .docx dominance means the model has limited experience with shorter, denser PDF text. Adding 200–300 more PDF examples from early-career and international candidates reduces format-based quality gaps at the data level.
Data-level fixes are always more robust than post-hoc corrections.
4
Add a minimum output floor for low-overlap inputs. When shared_token_ratio falls below 0.08, include an explicit system prompt instruction: "Even if vocabulary overlap is limited, generate a complete and rich tailored resume by identifying transferable skills and relevant experience."
Prevents career-changers from receiving near-empty outputs without requiring retraining.
5
Monitor output quality broken down by filetype and overlap band post-deployment. Track mean tailored_skill_item_count and tailored_experience_bullet_count separately for .pdf, .docx, and .doc users in production. A statistically significant persistent gap is a deployment red flag.
Fairness is not a one-time audit — it requires ongoing measurement. This is the difference between a responsible deployment and a compliant-on-paper one.
The broader ethical responsibility

The Raghavan et al. (2020) paper cited in this project's proposal makes a point directly relevant here: algorithmic hiring tools do not need to be intentionally discriminatory to cause discriminatory outcomes. A system that learned from historical resume data will encode the advantages of candidates who already knew how to write resumes in the dominant cultural format, in the dominant language, in the dominant file type. Our model is no exception.

The goal of reducing inference cost by 60% while maintaining output quality is a worthy engineering objective. But "maintaining output quality" must include asking: quality for whom? If the SLM produces outputs statistically indistinguishable from the teacher model on average, but systematically worse for non-native speakers and career-changers, we have not maintained quality — we have maintained the mean while widening the gap at the margins. The mitigations above are the difference between a resume optimizer that democratizes access to professional presentation and one that automates existing gatekeeping.

Section 08 Limitations of This Analysis
Honest acknowledgment of what this analysis can and cannot claim

Intellectual honesty requires stating clearly where the boundaries of this analysis lie. Three limitations are worth naming explicitly.

1
Proxy-based feature importance is an approximation, not an exact measurement. Because Qwen 4B is a generative model with no scalar prediction target, we cannot compute exact Shapley values. Our proxy approach — correlating engineered input features with output quality metrics across 1,530 samples — is the most defensible method available for this model type, but it captures association rather than causation. A feature that correlates with richer outputs may not be directly causing them; a third variable (e.g., overall resume quality) could be driving both. Future work could use perturbation-based methods — systematically varying individual features and measuring output changes — to move closer to causal attribution.
2
The bias audit is based on input feature distributions, not post-deployment demographic measurement. We identified subgroups at risk (career-changers, non-native English speakers, PDF users) based on what the EDA data tells us about how those groups differ on the features that drive model behavior. We have not measured actual output quality differences across real demographic groups in production — because the system has not yet been deployed and real user data does not exist. The audit is a risk map, not a confirmed harm report. Deployment must include the monitoring infrastructure described in Section 7 to convert this risk map into verified findings.
3
The training corpus of 1,530 pairs is sufficient for a research prototype but may underrepresent edge cases. The 3.79% outlier rate (58 resumes above 4,461 words) means the model has seen relatively few examples of very long resumes. Similarly, the 16.1% PDF representation and minimal presence of non-English or non-North-American resume formats means the model's behavior in those edge cases is less well-characterized than its behavior on the majority .docx format. Results and XAI findings should be interpreted with this distributional caveat in mind.
