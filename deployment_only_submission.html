Section 01 Production Readiness — Real-Time vs. Batch Inference
The core engineering challenge of this project is not building a model that works — it is building one that works at a price a real SaaS product can sustain. At 1,000 users each submitting 100 resume optimization requests per month, an API-based GPT-4 solution would cost approximately $8,000/month in inference alone — a figure that makes a viable product economically impossible at early-stage scale. Our self-hosted fine-tuned Qwen 4B brings that number down to $300–500/month, a reduction of more than 94%.
Achieving this requires a deliberate deployment architecture. The system operates in two distinct inference modes depending on who is asking and when they need an answer.
Real-time inference — primary user-facing mode

When a candidate submits their resume and a job description through the web interface, they expect a result within seconds — not minutes. The system handles this through synchronous, streaming inference on the self-hosted Qwen 4B model. The user sees the tailored resume being written token-by-token, which dramatically improves perceived responsiveness even when total generation time runs 15–40 seconds for a full 10K-token prompt.

At 4-bit quantization, the model requires only 2–3GB of VRAM, meaning a single A10G or RTX 4090 GPU can serve multiple concurrent users simultaneously — a critical requirement for SaaS scalability that the original API-based approach could never offer at this cost point.

Batch inference — secondary enterprise mode

Batch mode has two roles in this system. First, it was already used during data preparation: the four Gemini batch request files (521 + 521 + 523 + 1,044 requests) generated all 1,530 teacher outputs that the student model learned from — batch API pricing cut that data generation cost by approximately 40–50% versus real-time API calls.

In production, batch mode enables an enterprise recruiting use case: a firm wants to optimize 500 candidate resumes against a single job description overnight. All jobs queue to the same self-hosted GPU, process sequentially during off-hours, and return results by morning — at a marginal infrastructure cost near zero beyond the existing GPU rental.

Section 03 Monitoring & Maintenance — Preventing Model Staleness
A resume optimizer faces a particularly aggressive form of model drift because the job market vocabulary changes constantly. Skills like "prompt engineering" and "RAG pipelines" barely existed as resume keywords two years ago. A model trained on 1,530 pairs from 2024–2025 will gradually produce outputs that feel dated — missing emerging skills, over-representing obsolete ones, and misaligning with how job descriptions are written in 2026 and beyond. Monitoring is not optional maintenance; it is what keeps the 94% cost reduction economically defensible over time.
Data drift

Incoming resumes and job descriptions start looking different from your training distribution. New technology terms enter job postings; shared token ratio calculations become unreliable because the vocabulary the model learned to match is no longer representative of the current market.

Concept drift

The relationship between inputs and desired outputs changes even if inputs look similar. A "Senior Data Scientist" posting in 2025 emphasizes LLMs and vector databases; in 2023 it emphasized Spark and Hadoop. The model learned 2024 patterns and anchors outputs to those — not current market reality.

Output quality drift — the most dangerous type

Schema validity and hallucination rates remain stable, but semantic alignment scores quietly decline because the model's vocabulary no longer matches current job market language. This is invisible to structural validators — the JSON is valid, but the content is stale. Without active monitoring of semantic metrics, this drift goes undetected until users start complaining.

Retraining strategy — trigger-based, not schedule-based
Rather than retraining on a fixed calendar, the system uses three independent triggers. Any one of them reaching its threshold initiates a retraining cycle using only the QLoRA adapter weights — not the full Qwen 4B base model. Since adapter weights at rank r=16 are a tiny fraction of the full model, a retraining cycle runs on the same GPU used for inference during off-peak hours at near-zero additional cost.

Why staleness directly threatens the cost argument

The 94% cost reduction versus GPT-4 API only holds if the self-hosted SLM produces outputs of comparable quality. If the model drifts and quality degrades, users or the business may decide the gap justifies returning to the expensive API. A stale model producing mediocre outputs at $300/month is not a better product than a current model producing excellent outputs at $8,000/month. Monitoring and retraining is what makes the cost savings permanent rather than temporary.
